SECTION 01

問題診斷

網站在手機幾乎無法開啟，桌機載入也極為緩慢。 經過分析，根本原因來自以下三個問題同時存在：

01

縮圖未壓縮

226 張縮圖平均 130 KB，最大達 653 KB，總下載量高達 32 MB。即使有 loading="lazy"，首屏仍需下載大量資料。

02

Masonry 重排過多

每張圖片載入就觸發一次 masonry() 排版計算，頁面載入期間高達 200+ 次，手機 CPU 嚴重過載造成凍結。

03

YouTube 影片空白

53 個 <iframe> 在頁面載入時同時發起外部連線，且 file:// 本地協定被 YouTube 拒絕嵌入，影片一片空白。

FIX 01  ·  COMMIT 375e43a

縮圖批次壓縮

Python · Pillow

246 個縮圖以 quality=70 重新壓縮

對所有縮圖（*m.jpg、*m.png、*小.jpg、*s.jpg）批次重壓縮，最大寬度限制 600px。 PNG 無透明度者自動轉為 JPEG。原始高解析圖保留不動（僅 Fancybox 點擊後才下載）。

FIX 02  ·  COMMIT 375e43a

Masonry 排版 Debounce

JavaScript · setTimeout

200ms debounce，批次合併重排事件

原始程式碼每張圖片的 load 事件都直接呼叫 masonry()，200+ 張圖就觸發 200+ 次重排。 改用 debounce 後，200ms 內所有事件只執行一次排版，CPU 負擔降低 97%+。

// 修正前：每張圖載入都觸發（200+ 次）
$grid.imagesLoaded().progress(function() {
    $grid.masonry();  // ← 每次都執行
});

// 修正後：debounce，最多每 200ms 執行一次
var layoutTimer;
$grid.imagesLoaded().progress(function() {
    clearTimeout(layoutTimer);
    layoutTimer = setTimeout(function() {
        $grid.masonry('layout');
    }, 200);
});

FIX 04  ·  COMMIT 20aeaa6

點擊影片不跑版

CSS · padding-bottom 56.25%

16:9 固定容器，替換前後高度相同

原本用 replaceWith(iframe) 直接換掉 DOM 節點，iframe 高度（220px）與縮圖高度不符，Masonry 重算後整頁跑版。 改用 padding-bottom: 56.25% 固定 16:9 比例容器，iframe 以絕對定位填滿，替換前後容器高度完全一致。

/* 固定 16:9 容器，點擊前後高度不變 */
.yt-facade, .yt-playing {
  position: relative;
  padding-bottom: 56.25%;  /* 16:9 */
}
.yt-playing iframe {
  position: absolute;
  top: 0; left: 0;
  width: 100%; height: 100%;
}
FIX 05  ·  2026-04-14

Git LFS 遷移

Git LFS · git-filter-repo

647 個大型檔案移出 git 歷史，push 從分鐘級降至秒級

每次 git push 需傳輸整個 pack 檔（623 MB），原因是所有圖片（jpg/png/pdf/obj）的歷史版本都壓縮在 git 物件庫中。 透過 git lfs migrate import --everything 將全部二進位大檔改由 LFS 管理， git history 只保留小型指標文字（每個約 130 bytes），實際檔案存放於 GitHub LFS 伺服器。
